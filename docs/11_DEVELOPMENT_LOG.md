## 2026-03-16 - 최종 문서 개정안 확정 반영

### 반영 목적

- 기존 선수 중심 프로필 구조를 공통 사용자 구조로 정리한다.
- player / referee / facility_manager 다중 역할 구조를 문서에서 먼저 확정한다.
- 시설 관리자 구조를 시설과의 관계 모델로 정리한다.
- 베트남 운영 기준 지역 구조를 3단계 코드 기반으로 확장한다.
- 다국어 저장, Settings 메뉴, wallet/payment 구조를 문서에 선반영한다.

### 핵심 반영 내용

- `profiles`를 공통 사용자 정보 전용 구조로 재정의했다.
- `account_types`를 도입해 한 사용자가 여러 서비스 역할을 가질 수 있도록 문서화했다.
- `player_profiles`, `referee_profiles`를 역할별 상세 구조로 분리했다.
- 시설 관리자 구조는 `facilities + facility_managers` 관계 모델을 우선 채택했다.
- 지역 구조는 `country_code`, `province_code`, `district_code` 3단계 코드로 확정했다.
- 지역 입력은 자유 입력이 아니라 내부 정적 데이터 또는 seed 데이터 기반 cascading select를 사용하도록 정리했다.
- `preferred_language`는 디바이스 언어 감지 우선, 실패 시 `vi`, 이후에는 서버 저장값 우선 규칙으로 정리했다.
- `wallet_accounts.balance`는 표시값이며 정합성의 기준은 `wallet_transactions`라는 원칙을 명시했다.
- `payment_intents` 상태값과 상태 전이 규칙을 정리했다.
- 결제 확정은 클라이언트가 아니라 서버 검증 경로만 허용하도록 정리했다.
- Settings 메뉴 구조와 Phase 2 MVP에서 실제 활성화할 항목 범위를 구분했다.
- Expo Router 라우팅 그룹 구조를 온보딩, 메인 탭, 설정, 시설, 관리자 기준으로 정리했다.

### 이번 반영 범위

- 문서 구조 개정만 수행
- 실제 DB 스키마 변경 및 서버 함수 구현은 후속 작업

- `primary_region_code`는 즉시 삭제하지 않고 deprecated 상태로 유지했다.

- 기존 유저에 `player` 역할을 기본 부여하는 백필 SQL을 작성했다.
- 기존 `profiles` 선수 필드를 `player_profiles`로 이관하는 백필 SQL을 작성했다.
- `province_code`, `district_code`, 선수 필드 not null 강화는 이번 단계에서 미뤘다.
- RLS는 다음 단계로 분리하고 이번에는 준비용 테이블 상태까지만 마무리했다.
